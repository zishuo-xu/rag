# RAG 知识库问答系统需求文档

## 1. 文档信息

**项目名称：** 企业知识库智能问答系统（RAG）  
**文档版本：** V1.0  
**文档类型：** 产品/需求说明书（PRD）  
**适用对象：** 开发、测试、面试展示、项目复盘  
**项目目标：** 构建一个支持文档上传、解析、切分、向量化、检索、问答和引用返回的知识库问答系统，帮助用户基于已有资料进行准确问答，降低大模型幻觉，提高企业内部知识检索与问答效率。

---

## 2. 项目背景

随着大语言模型在企业场景中的普及，越来越多的业务希望让模型基于企业自己的文档进行回答，而不是仅依赖模型通用知识。但直接让模型回答存在以下问题：

1. 无法访问企业内部知识；
2. 容易产生幻觉；
3. 难以追溯答案来源；
4. 缺乏可控性与工程落地能力。

因此，需要构建一套基于 RAG（Retrieval-Augmented Generation，检索增强生成）的知识库问答系统。系统先从企业文档中检索相关内容，再将检索结果与用户问题一同提交给大模型生成答案，从而提升回答的准确率、可解释性和可用性。

---

## 3. 项目目标

### 3.1 业务目标

1. 支持用户上传知识文档并建立知识库；
2. 支持用户基于知识库进行自然语言提问；
3. 系统能够返回基于文档内容生成的答案；
4. 系统能够返回答案对应的文档引用，增强可信度；
5. 具备基础工程能力，便于后续扩展为多用户、多知识库、多租户版本。

### 3.2 技术目标

1. 实现完整 RAG 链路：上传 -> 解析 -> 切分 -> 向量化 -> 检索 -> 重排 -> 生成；
2. 建立可维护的后端服务结构；
3. 提供标准化接口，支持前端接入；
4. 支持基础日志、错误处理和状态管理；
5. 为面试展示提供清晰的系统架构和模块边界。

---

## 4. 项目范围

### 4.1 本期范围（V1）

1. 支持单用户使用；
2. 支持上传 txt、md、pdf、docx 格式文档；
3. 支持文档解析与文本切分；
4. 支持向量化入库；
5. 支持基于知识库提问；
6. 支持返回答案与引用来源；
7. 支持基础文档管理；
8. 支持简单问答历史查询。

### 4.2 暂不纳入范围

1. OCR 扫描件识别；
2. 多租户复杂权限体系；
3. 复杂前端设计；
4. 复杂工作流 Agent；
5. 文档协同编辑；
6. 自动数据标注与模型微调；
7. 在线知识更新冲突处理。

## 5. 用户角色

### 5.1 普通用户

核心诉求：
- 上传自己的知识文档；
- 向系统提问；
- 获取基于文档的准确答案；
- 查看答案引用来源。

### 5.2 管理员（可选扩展）

核心诉求：
- 管理知识库文档；
- 删除异常文档；
- 查看系统运行状态。

V1 可弱化管理员能力，以系统维护者身份存在。

## 6. 使用场景

### 6.1 场景一：企业制度问答
用户上传公司报销制度、请假制度、出差规范等文档，随后提问“差旅报销标准是什么”，系统返回准确答案及引用内容。

### 6.2 场景二：技术文档问答
用户上传系统设计文档、接口说明、部署文档，提问“支付服务超时时间配置在哪里”，系统从技术文档中检索并回答。

### 6.3 场景三：产品 FAQ 问答
用户上传产品说明书、售后 FAQ 等文档，提问“退款申请多久处理完成”，系统返回相关流程说明。

---

## 7. 核心流程

### 7.1 文档入库流程
1. 用户上传文档；
2. 系统保存原始文件与元数据；
3. 系统解析文档文本；
4. 系统对文本进行切分；
5. 系统调用 embedding 服务将 chunk 向量化；
6. 系统将 chunk 文本、向量和元数据写入向量存储；
7. 系统更新文档状态为“已完成”。

### 7.2 问答流程
1. 用户选择知识库或默认知识库；
2. 用户输入问题；
3. 系统对问题生成 embedding；
4. 系统执行向量检索或混合检索；
5. 系统对召回结果进行重排；
6. 系统组装 Prompt，将问题和检索内容提交给大模型；
7. 系统返回答案、引用来源与相关片段；
8. 系统记录问答日志。

## 8. 功能需求

## 8.1 文档管理模块

### 8.1.1 文档上传
**功能描述：** 用户上传知识文档，系统接收并保存。

**输入：** 文件、文件名、文件类型  
**输出：** 上传结果、文档 ID、处理状态

**功能要求：**
1. 支持 txt、md、pdf、docx 格式；
2. 单个文件大小限制可配置，例如 20MB；
3. 上传成功后返回文档 ID；
4. 初始状态为 `PROCESSING`；
5. 非法格式需提示错误。

### 8.1.2 文档列表查询
**功能描述：** 查询已上传文档及处理状态。

**展示字段建议：**
- 文档 ID
- 文件名
- 类型
- 上传时间
- 状态（处理中/成功/失败）
- chunk 数量

### 8.1.3 文档删除
**功能描述：** 删除指定文档及关联 chunk、向量数据。

**功能要求：**
1. 删除文档元数据；
2. 删除原始文件；
3. 删除对应向量记录；
4. 删除失败需回滚或记录异常。

---

## 8.2 文档解析模块

### 8.2.1 文档解析
**功能描述：** 将原始文档解析为纯文本内容。

**功能要求：**
1. txt、md 直接读取文本；
2. pdf 提取文本型内容；
3. docx 提取正文文本；
4. 对空文档、乱码文档、解析失败文档做异常处理；
5. 记录解析失败原因。

### 8.2.2 文档清洗
**功能描述：** 对解析后的文本进行基础清洗。

**清洗内容：**
1. 去除多余空行；
2. 标准化换行符；
3. 去除明显无意义字符；
4. 保留必要标题层级信息。

---

## 8.3 文本切分模块

### 8.3.1 切分策略
**功能描述：** 将长文本切分为适合检索与生成的小块。

**功能要求：**
1. 支持固定长度切分；
2. 支持 overlap 重叠；
3. 每个 chunk 保留顺序编号；
4. 保留来源文档 ID；
5. 可扩展支持按标题/段落切分。

**默认策略建议：**
- chunk 长度：500~800 字符
- overlap：50~100 字符

### 8.3.2 chunk 元数据
每个 chunk 至少应包含：
- chunk_id
- document_id
- chunk_index
- chunk_text
- section_title（可选）
- page_no（可选）
- created_time

---

## 8.4 向量化模块

### 8.4.1 文档向量化
**功能描述：** 将 chunk 文本转为 embedding 向量并存储。

**功能要求：**
1. 支持批量向量化；
2. 向量化失败时记录失败原因；
3. 向量维度由模型决定；
4. 保存向量与 chunk 关联关系。

### 8.4.2 问题向量化
**功能描述：** 将用户问题转为向量，用于相似度检索。

---

## 8.5 检索模块

### 8.5.1 语义检索
**功能描述：** 基于问题向量从向量库召回相似 chunk。

**功能要求：**
1. 支持 topK 配置；
2. 返回相似度分数；
3. 支持按文档范围过滤（可选）；
4. 响应时间可接受。

### 8.5.2 混合检索（V1.1 可选）
**功能描述：** 同时使用关键词检索与向量检索，提高召回稳定性。

### 8.5.3 结果重排
**功能描述：** 对召回结果进行二次排序，提高上下文质量。

**功能要求：**
1. 支持保留前 N 条高质量结果；
2. 支持按相关性重新排序；
3. 便于后续替换为独立 rerank 模型。

---

## 8.6 问答生成模块

### 8.6.1 Prompt 组装
**功能描述：** 将系统提示词、用户问题和检索结果组装为模型输入。

**要求：**
1. 明确要求模型基于提供资料回答；
2. 若资料不足，需明确说明无法确定；
3. 要求输出引用来源；
4. 限制模型随意扩展。

### 8.6.2 模型调用
**功能描述：** 调用大模型生成答案。

**功能要求：**
1. 支持调用外部模型接口；
2. 支持超时控制；
3. 支持错误重试；
4. 记录调用耗时和状态；
5. 可扩展支持多模型切换。

### 8.6.3 答案返回
**输出结构建议：**
- question
- answer
- citations
- related_chunks
- request_id
- elapsed_time

---

## 8.7 引用与可解释性模块

### 8.7.1 引用返回
**功能描述：** 在答案中提供文档来源信息。

**功能要求：**
1. 返回命中的文档名；
2. 返回命中的 chunk 内容摘要；
3. 返回文档位置（页码/段落编号可选）；
4. 支持前端点击查看原文片段。

### 8.7.2 拒答策略
**功能描述：** 当知识库中无足够信息时，系统提示“知识库暂无足够依据”。

**目标：**
- 降低幻觉；
- 提升可信度；
- 强调基于资料回答。

---

## 8.8 问答记录模块

### 8.8.1 记录问答历史
**记录内容建议：**
- request_id
- question
- answer
- top_chunks
- create_time
- response_time
- model_name
- status

### 8.8.2 查询问答历史
V1 可仅支持按时间倒序查询最近若干条记录。

## 9. 非功能需求

### 9.1 性能要求
1. 单文档上传后处理时间可接受；
2. 单次问答响应时间目标控制在数秒内；
3. 检索与模型调用需分阶段记录耗时。

### 9.2 可用性要求
1. 文档处理失败时应有明确错误提示；
2. 问答失败时应返回统一错误码；
3. 模型不可用时系统应提示重试。

### 9.3 可扩展性要求
1. 支持后续替换 embedding 模型；
2. 支持后续替换向量库；
3. 支持后续增加多知识库；
4. 支持后续增加权限管理。

### 9.4 安全性要求
1. 限制上传格式；
2. 对上传文件名进行校验；
3. 避免非法脚本文件伪装上传；
4. 记录关键操作日志。

### 9.5 可维护性要求
1. 模块边界清晰；
2. 日志可追踪；
3. 配置项可外部化；
4. 接口命名清晰一致。

## 11. 接口设计建议

## 11.1 上传文档
**POST** `/api/documents/upload`

请求：multipart/form-data  
响应：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "documentId": 123,
    "status": "PROCESSING"
  }
}
```

## 11.2 查询文档列表
**GET** `/api/documents`

## 11.3 删除文档
**DELETE** `/api/documents/{id}`

## 11.4 发起问答
**POST** `/api/qa/ask`

请求示例：
```json
{
  "question": "退款规则是什么？",
  "documentIds": [123, 124]
}
```

响应示例：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "requestId": "req_001",
    "answer": "根据知识库内容，退款申请将在 3 个工作日内处理完成。",
    "citations": [
      {
        "documentId": 123,
        "fileName": "售后说明书.pdf",
        "chunkIndex": 4,
        "content": "退款申请提交后，系统将在 3 个工作日内审核处理。"
      }
    ]
  }
}
```

## 11.5 查询问答记录
**GET** `/api/qa/history`

- FAILED：处理失败
- DELETED：已删除

### 12.2 问答状态
- SUCCESS：成功
- FAILED：失败

---

## 13. 异常处理

### 13.1 上传异常
- 文件格式不支持
- 文件过大
- 文件为空

### 13.2 解析异常
- pdf 内容为空
- docx 解析失败
- 字符编码异常

### 13.3 向量化异常
- embedding 接口失败
- 向量服务超时

### 13.4 问答异常
- 检索无结果
- 模型调用失败
- 请求超时

系统需统一返回错误码与错误信息，并记录日志。

---

## 14. 验收标准

### 14.1 文档侧验收
1. 支持上传至少一种文本型文档；
2. 文档可成功解析、切分并入库；
3. 文档状态可查询。

### 14.2 问答侧验收
1. 用户可输入问题并得到回答；
2. 回答需基于知识库内容；
3. 返回至少一个引用来源；
4. 对知识库未覆盖问题可进行拒答。

### 14.3 工程侧验收
1. 接口可正常调用；
2. 数据库存储结构清晰；
3. 日志中可追踪一次请求流程；
4. 文档删除后相关数据同步清理。

## 16. 技术方案建议（面向 Java 程序员）

推荐技术栈：
- 后端：Spring Boot
- 数据库：PostgreSQL
- 向量能力：pgvector
- 文件存储：本地文件系统 / MinIO
- 模型调用：外部 LLM API
- 解析组件：Apache Tika / PDFBox / POI

架构建议：
1. Controller：接口层
2. Service：业务处理层
3. Parser：文档解析层
4. Splitter：文本切分层
5. EmbeddingClient：向量化调用层
6. RetrievalService：检索层
7. LlmClient：模型调用层
8. Repository：数据访问层

---

## 17. 风险与注意事项

1. 文档解析质量会直接影响检索效果；
2. chunk 切分策略不合理会导致回答效果差；
3. 向量检索可能召回不稳定，需要后续增加 rerank；
4. 大模型若不限制 Prompt，容易产生幻觉；
5. 若没有引用返回，系统可信度不足；
6. 若只讲“调用 API”，项目含金量会下降，需要强调工程设计。
